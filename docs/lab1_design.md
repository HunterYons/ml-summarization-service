# Лабораторная работа №1: Определение задачи и анализ данных

---

### 1. Предметная область

Область исследования — обработка естественного языка (NLP), а именно задача **Abstractive Summarization** (абстрактивная суммаризация). В отличие от экстрактивной суммаризации, которая копирует предложения из оригинала, выбранный подход генерирует новый текст, перефразируя основное содержание.

---

### 2. Цели и задачи проекта

**Бизнес-цель:** Сократить время пользователя на ознакомление с длинными русскоязычными текстами (новости, статьи, отчёты) при сохранении ключевого смысла.

**Целевая аудитория:** Пользователи Telegram, работающие с новостными статьями на русском языке объёмом до ~800 слов.

**Технические задачи:**
- Разработка микросервиса с REST API (FastAPI) и Telegram-ботом для взаимодействия с пользователем.
- Инференс предобученной нейросетевой модели (T5) в контейнеризированной среде.
- Внедрение системы трекинга метрик качества через MLflow.

---

### 3. Функциональные требования (ФТ)

| ID  | Требование |
|-----|------------|
| ФТ-1 | Система принимает русскоязычный текст через Telegram-бота или REST API. |
| ФТ-2 | Система возвращает краткое содержание текста (абстрактивная суммаризация). |
| ФТ-3 | Результат содержит метрики качества: Semantic Similarity и Latency. |
| ФТ-4 | Все запросы логируются в MLflow и сохраняются в PostgreSQL. |
| ФТ-5 | Бот обрабатывает команды `/start`, `/help`, `/history`. |

---

### 4. Нефункциональные требования (НФТ)

| ID   | Требование | Значение |
|------|------------|----------|
| НФТ-1 | **Ограничение длины входного текста** | Максимум **1024 токена** (~700–800 слов). Текст, превышающий этот предел, обрезается токенизатором (`truncation=True`). Пользователь об этом не предупреждается — это известное ограничение модели T5. |
| НФТ-2 | **Время ответа (Latency)** | Не более 15 секунд на CPU для текста длиной до 1024 токенов. |
| НФТ-3 | **Минимальная длина входного текста** | Не менее 10 символов (валидация на уровне бота). |
| НФТ-4 | **Язык** | Только русский язык. Модель `rut5_base_sum_gazeta` обучена исключительно на русскоязычных новостях и даёт некорректные результаты для других языков. |
| НФТ-5 | **Доступность** | Сервис работает локально; горизонтальное масштабирование не предусмотрено в текущей версии. |
| НФТ-6 | **Тип входных текстов** | Новостные статьи и официальные тексты в новостном стиле. Художественная литература, диалоги и технические документы — вне области применения модели. |

> **Ключевое ограничение архитектуры:** Поскольку система целиком опирается на предобученную модель T5 без дообучения, качество суммаризации определяется распределением обучающих данных самой модели (корпус Gazeta.ru). Тексты, стилистически далёкие от новостного жанра, могут давать результаты ниже целевых метрик.

### 5. Исследуемые данные

В качестве эталонного набора для оценки качества используется датасет **[IlyaGusev/gazeta](https://huggingface.co/datasets/IlyaGusev/gazeta)** — корпус новостей Gazeta.ru.

- **Входные данные (text):** Тексты новостных статей на русском языке.
- **Целевая переменная (summary):** Краткая аннотация, написанная профессиональными редакторами.
- **Разбивка датасета:** train / validation / test.

#### 5.1. Анализ длин текстов

Перед выбором ограничения в 1024 токена был проведён анализ распределения длин статей в тестовой выборке датасета (`test`, 500 примеров). Длина измерялась токенизатором модели `rut5_base_sum_gazeta`.

| Статистика | Значение |
|---|---|
| Медиана длины текстов | ~620 токенов |
| 75-й перцентиль | ~910 токенов |
| 90-й перцентиль | ~1 180 токенов |
| Доля текстов ≤ 1024 токенов | **~79%** |
| Доля текстов > 1024 токенов (обрезаются) | **~21%** |

**Вывод:** Примерно каждый пятый текст в датасете превышает лимит модели и будет обрезан. Это зафиксировано в НФТ-1. При необходимости работы с длинными текстами следует рассматривать стратегии разбивки (chunking) или использование модели с большим контекстным окном.

### 6. Метрики качества

Для оценки качества суммаризации выбраны три взаимодополняющие метрики:

| Метрика | Что измеряет | Целевое значение |
|---|---|---|
| **ROUGE-1 / ROUGE-2 / ROUGE-L** | Лексическое перекрытие между результатом и эталонной аннотацией из датасета | ROUGE-1 > 0.30 |
| **Semantic Similarity (SBERT)** | Косинусное сходство эмбеддингов оригинала и суммаризации — сохранение смысла при перефразировании | > 0.60 |
| **Inference Latency** | Время генерации одного ответа | < 15 сек на CPU |

> **Важно:** Метрика Semantic Similarity вычисляется относительно **входного текста**, а не эталонной аннотации. Это не оценка качества суммаризации, а оценка того, насколько суммаризация «близка по смыслу» к источнику. Для честной оценки качества необходим ROUGE относительно эталонных аннотаций датасета.

### 7. Baseline без ML

Перед применением тяжёлой модели T5 необходимо установить **нижнюю планку качества** с помощью простых алгоритмических подходов. Это нужно, чтобы обосновать целесообразность использования нейросети.

Используются два baseline:

**Baseline 1 — Lead-2 (экстрактивный):**
Берутся первые два предложения текста. Это стандартный baseline для новостных текстов, так как журналисты, как правило, выносят главное в лид.

**Baseline 2 — TextRank (экстрактивный):**
Алгоритм на основе графа предложений, не требующий обучения. Реализован через библиотеку `sumy`.

Результаты baseline на тест-сете (500 примеров) для сравнения с T5 приведены в Лабораторной работе №3.
