# Лабораторная работа №4: Развертывание и промышленный мониторинг (MLOps)

### 1. Цель работы
Перевод ML-модели из состояния эксперимента в состояние готового сервиса. Внедрение практик контейнеризации для воспроизводимости среды и системы мониторинга для контроля качества работы модели в «дикой» среде (production).

### 2. Оркестрация микросервисов (Docker Compose)
Система спроектирована как распределенное приложение. Изоляция компонентов позволяет обновлять модель или интерфейс независимо друг от друга.

**Контейнерная инфраструктура:**
* **api:** Сердце системы (FastAPI). Выполняет инференс T5 и расчет семантических метрик.
* **bot:** Асинхронный клиент (Aiogram). Обрабатывает ввод пользователя, следит за соблюдением НФТ-3 (минимальная длина).
* **mlflow:** Сервер трекинга. Позволяет визуализировать результаты инференса.
* **db:** PostgreSQL. Обеспечивает персистентность данных для MLflow.

### 3. Мониторинг и наблюдаемость (Observability)
Для решения проблемы «калибровки метрик», о которой говорилось в ЛР №1, в MLflow реализован расширенный логинг каждого запроса. 

**Логируемые параметры в реальном времени:**
1.  **Input Length:** Позволяет отслеживать долю текстов, попавших под `truncation` (согласно НФТ-1).
2.  **Compression Ratio:** Отношение длины суммаризации к оригиналу. Помогает понять, не слишком ли модель «ужимает» текст.
3.  **Semantic Similarity (SBERT):** Основной индикатор того, что модель не «галлюцинирует» и сохраняет смысл.
4.  **Inference Latency:** Контроль соблюдения НФТ-2 (не более 15 сек).



### 4. Обработка ограничений в эксплуатации
В соответствии с функциональным дизайном, система обрабатывает пограничные случаи:
* **Длинные тексты (>1024 токена):** Система выполняет обрезку на уровне API. В MLflow такие запросы помечаются для дальнейшего анализа: если Similarity падает ниже 0.60 на обрезанных текстах, это сигнал к внедрению стратегии Chunking (разбиение на части).
* **Холодный запуск:** Реализованы Docker Healthchecks, которые не позволяют Telegram-боту принимать сообщения, пока веса модели T5 (~1 ГБ) не будут полностью загружены в RAM видеокарты или процессора.

### 5. Сравнение с Baseline в реальном времени
Архитектура API позволяет параллельно с моделью T5 вычислять результат **Lead-2** (базовая линия из ЛР №3). Это позволяет проводить «теневое тестирование»: мы всегда видим, насколько нейросеть выигрывает у простого алгоритма на реальных данных пользователей.

### 6. Результаты внедрения пайплайна
* **Воспроизводимость:** Полное развертывание стека занимает < 5 минут одной командой `docker-compose up`.
* **Контроль качества:** Благодаря MLflow выявлено, что среднее семантическое сходство на реальных новостных запросах составляет **0.72-0.75**, что соответствует целевым показателям из ЛР №1.
* **Отказоустойчивость:** Разделение на API и Бот позволяет сервису не «падать» целиком при перегрузке вычислительного модуля.

### 7. Заключение по курсу
В ходе выполнения цикла лабораторных работ была реализована полноценная ML-система. Проект прошел путь от анализа распределения длин в датасете `gazeta` и выбора метрик до создания отказоустойчивой микросервисной архитектуры с профессиональным мониторингом. Система готова к масштабированию и замене модели на более мощные архитектуры без изменения инфраструктурного кода.
