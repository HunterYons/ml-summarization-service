# Лабораторная работа №2: Архитектура и стратегия валидации

### 1. Стратегия валидации данных
Для обеспечения качества модели используется стратегия **Hold-out validation** с распределением данных 80/10/10 (обучение/валидация/тест).

* **Предобработка:** Перед подачей текста в модель `T5` выполняется очистка от специфических символов разметки и нормализация пробелов.
* **Контроль утечек (Leakage Prevention):** В пайплайн встроена проверка на дубликаты. Валидационный набор не содержит текстов, которые использовались при дообучении (fine-tuning) оригинальной модели `rut5_base_sum_gazeta`.
* **Оценка в реальном времени:** Каждая суммаризация, выполненная через API, проходит автоматическую оценку семантического сходства (Semantic Similarity) с оригиналом для мониторинга «галлюцинаций» модели.

### 2. Производительность и масштабируемость
В архитектуре реализованы следующие решения для оптимизации работы на CPU:

* **Truncation & Padding:** Ограничение входной последовательности до **1024 токенов**. Это позволяет избежать переполнения оперативной памяти (OOM) и обеспечивает стабильное время ответа.
* **Async/Await + ThreadPool:** Поскольку инференс нейросети — это CPU-интенсивная задача, в FastAPI реализовано разделение потоков. Основной Event Loop обрабатывает сетевые запросы, а вычисления модели выносятся в отдельный пул потоков, что предотвращает блокировку API.
* **Batch Processing:** Поддержка пакетной обработки текста (batching) на уровне API для эффективного использования ресурсов процессора при высокой нагрузке.

### 3. Технологический стек (Микросервисы)
Система развернута в изолированных Docker-контейнерах:

1.  **Backend (FastAPI):**
    * Обработка POST-запросов на суммаризацию.
    * Расчет метрик (ROUGE-L, Cosine Similarity).
    * Интеграция с MLflow Tracking Server.
2.  **Frontend (Aiogram 3.x):**
    * Асинхронный Telegram-бот.
    * Валидация длины сообщения (минимум 10 символов).
    * Обработка ошибок сети и таймаутов API.
3.  **ML Ops (MLflow):**
    * Сбор телеметрии.
    * Хранение истории инференса и значений точности.
4.  **Database (PostgreSQL):**
    * Хранение метаданных экспериментов.



### 4. Описание архитектуры API
API спроектировано по принципу **REST**. Основной эндпоинт `/summarize` принимает JSON-объект и возвращает результат вместе с вычисленными метриками.

**Схема потока данных (Data Flow):**
1.  Бот получает текст от пользователя -> Отправляет асинхронный запрос к API.
2.  API токенизирует текст -> Выполняет генерацию (модель T5) -> Считает эмбеддинги (модель SBERT).
3.  API логирует параметры запроса и метрики в MLflow.
4.  API возвращает результат боту -> Бот выводит текст и метрику сходства пользователю.

### 5. Инфраструктурные решения
* **Containerization:** Весь проект описывается файлом `docker-compose.yml`, что гарантирует идентичность среды разработки и эксплуатации.
* **Health Checks:** Настроены проверки состояния сервисов (Healthchecks), чтобы бот не начинал работу до полной загрузки весов нейросети в память API.
